# 설계 원칙 — 정직함 & 어포던스

측정·알림·UI 를 결정할 때 우선하는 원칙. 심리·생리적 근거는 [affective-design.md](./affective-design.md),
문구·시각 마감은 [visual-polish.md](./visual-polish.md) 참고.

## 1. 정직함 (Honesty)

- **측정값은 추정치**: 이 앱은 입력 유무만 알 수 있다(마지막 입력 시각 기반). 사용자 노출 문구는 "추정치"로 표현하고 단정하지 않는다(README·도움말·면책).
- **한계를 숨기지 않는다**: 위장 모드는 이 측정 방식의 한계(하드웨어/합성 입력 미구분)를 직접 보여주는 실험 기능임을 명시하고, 켤 때 책임 고지에 동의를 받는다.
- **거짓 피드백 금지**: 실제 상태와 다른 표시를 하지 않는다. 툴팁의 `[정지]`/`[위장]`, 메뉴 체크 상태는 항상 실제 상태와 동기화한다.

## 2. 어포던스 (Affordance)

지각된 어포던스·시그니파이어·피드백·매핑(Norman).

- **즉각 피드백**: 모든 조작(일시정지, 설정 저장/다시 읽기, 위장 토글)에 곧바로 보이는 반응 — 풍선·툴팁·체크 상태·아이콘 색.
- **제약(constraints)으로 실수 방지**: 오해 소지가 있는 조작(위장 모드 켜기/끄기)은 확인 팝업을 거친다. 첫 실행 동의는 체크해야 버튼이 활성화된다.
- **정직한 매핑**: 설정 창의 값 = `config.ini` = 실제 동작. 저장하면 바로 적용된다.

## 3. 알림 윤리 (사용자 통제권)

알림은 돕는 도구지 잔소리가 아니다.

- 모든 알림은 **개별 토글 + 마스터 토글**로 끌 수 있고, 수면시간에는 전부 무음이다.
- 반복 알림은 **1시간 간격으로 절제**하고, 밀린 알림을 한꺼번에 울리지 않는다(`PrimeNotifiedFlags`·카운터 점프 — [DEVELOPMENT.md](./DEVELOPMENT.md) "알림").
- 죄책감·압박을 유발하는 문구 금지 — 권유형으로 쓴다("잠깐 쉬어가는 건 어때요?"). 문구 규칙은 [visual-polish.md](./visual-polish.md) §1.

## 4. 충돌 시 우선순위

원칙끼리 부딪히면 이 순서로 판단한다.

1. **정직함 > 감성·편의** — 연출·문구가 사실을 왜곡하게 되면 그 기법을 수정하거나 버린다.
2. **사용자 통제권 > 기능 어필** — 알림을 늘려 존재감을 키우기보다, 언제든 끌 수 있게 만든다.
3. **가용성 > 데이터 완결성** — 파일 I/O 실패로 앱이 죽지 않게 한다. 단, 부분 데이터로 원본을 덮어쓰는 것은 완결성 침해이므로 그 회차 저장을 건너뛴다([DEVELOPMENT.md](./DEVELOPMENT.md) "코딩 컨벤션").

**적용 사례**: 위장 모드는 '측정을 속이는 기능'이라 정직함과 정면 충돌한다 → 정식 기능이 아닌 **테스트(실험) 기능**임을 명시하고, 켜고 끌 때마다 확인 팝업을 거치며, **자기 자신의 측정값도 함께 오염됨**을 고지하는 것으로 해소했다. 속임수를 팔지 않고, 측정 방식의 한계를 보여주는 데모로 자리매김한 것.

## 5. 새 기능 점검 체크리스트

기능·알림을 추가하기 전에 확인한다.

- [ ] 사용자 노출 문구에 기술 용어가 없고, 측정값을 단정하지 않는가(추정치)
- [ ] 조작의 결과가 화면(툴팁·체크 상태·풍선)에 즉각·정직하게 반영되는가
- [ ] 실수하기 쉽거나 오해 소지가 있는 조작이라면 제약(확인 팝업·비활성화)을 뒀는가
- [ ] 알림이라면 개별 토글·마스터 토글·수면시간을 모두 존중하는가(`Notify()` 경유)
- [ ] 사용자가 끌 수 있는가(통제권) — 끌 수 없는 알림·연출은 만들지 않는다
