# UI·문구 마감

사용자에게 보이는 모든 것(창·알림 문구·문서)의 마감 규칙. 원칙은 [design-principles.md](./design-principles.md), 색·시점의 근거는 [affective-design.md](./affective-design.md).

## 1. 문구 톤

- 한국어 존댓말, 짧고 따뜻하게. **명령이 아닌 권유** — "잠깐 쉬어가는 건 어때요?", "무리하지 마세요".
- 죄책감·압박을 유발하는 표현 금지(알림 윤리 — design-principles §3).
- 기술 용어(`GetLastInputInfo` 등)는 사용자 노출 문구에 쓰지 않는다 — "추정치"로 표현. 기술 설명은 개발 문서에만.
- 과장·세일즈성 문구 금지. 사실을 담백하게.

| 상황 | ⛔ 이렇게 쓰지 않는다 | ✅ 현재 스타일 |
|---|---|---|
| 휴식 권유 | "1시간 넘게 쉬지 않았습니다. 휴식하십시오." | "1시간 0분 연속 근무 중이에요. 잠깐 쉬어가는 건 어때요?" |
| 야간 근무 | "야간 근무 경고!" | "야간 시간대예요. 무리하지 마세요." |
| 정시 퇴근 | "근무 종료." | "정규 근무시간이 끝났어요. 오늘 실근무 7시간 30분 — 수고하셨습니다!" |

## 2. 문서 표기

- 시간 범위는 물결표(`~`) 대신 **엔대시(`–`)**: `00:00–07:00`. 한 줄에 물결표 범위가 두 개면 GFM이 취소선으로 렌더한다([DEVELOPMENT.md](./DEVELOPMENT.md) "배운 것").
- 사용자 문서(README)는 기능 나열보다 **사용 시나리오 중심**으로.

## 3. 시각

- **제한된 팔레트**: 히트맵은 회색(0) → 옅은 파랑 → 진한 파랑(1시간+)의 sqrt 스케일 단일 색상. 원색 남발 금지.
- 폰트는 시스템 기본(맑은 고딕·Segoe UI)으로 일관 — 커스텀 폰트 짜깁기보다 기본의 일관성이 낫다.
- 창은 고정 크기 대화상자(`FixedDialog`/`FixedSingle`). 요소는 **점유 좌표를 계산해 겹침 없이** 배치한다.
- **설정 창 레이아웃 관례**(`SettingsForm` — 새 행도 이 수치를 따른다):
  - 라벨 x=16(폭 170) / 입력 컨트롤 x=196(폭 130) / 행 간격 30px
  - 그룹박스 폭 348 / 창 내용 폭 372 / 체크박스 2열은 x=28·x=188
  - 버튼은 `FlatStyle.System`(네이티브 룩 — 커스텀 그리기보다 OS 일관성)

## 4. 트레이 알림·툴팁 제약

- **툴팁(`NotifyIcon.Text`)은 63자 제한** — 초과분은 잘린다(`UpdateTooltip` 방어 코드 있음). 새 항목을 넣을 땐 총 길이를 계산할 것.
- 풍선 알림 제목은 **짧은 명사구**("휴식이 필요해요", "정시 퇴근 시간"), 본문은 1–2문장.
- 표시 시간: 상태 알림 5초(`Notify`), 조작 피드백 2–2.5초.
- 풍선은 OS 스타일이라 색·폰트를 제어할 수 없다 — **감정 톤은 문구와 아이콘(Info/Warning)으로만** 전달한다. Warning 은 경고성(야간·초과·휴식)에만.

## 5. 검증

- UI 변경은 렌더 하니스로 PNG 를 뽑아 눈으로 확인한다([DEVELOPMENT.md](./DEVELOPMENT.md) "UI 렌더 검증 하니스").

## 체크리스트 (사용자 노출 변경 시)

- [ ] 새 문구가 권유형·존댓말이고 기술 용어가 없는가
- [ ] 시간 범위 표기가 엔대시(–)인가
- [ ] 새 UI 요소가 레이아웃 관례(16/196/행 30px)를 따르고 기존 요소와 겹치지 않는가
- [ ] 툴팁이 63자 안에 드는가
- [ ] 렌더 하니스 PNG 로 확인했는가
